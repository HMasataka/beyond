# ADR-0001: タスク管理の構造（マイルストーン / 親 issue / 子 issue）

- ステータス: Accepted
- 日付: 2026-06-16

## 背景

タスクには性質の異なる 2 つの軸がある。

- 構造軸: どのコンポーネントの話か（transport / runtime / client …）。コンポーネントという観点は恒久だが、それを束ねる親 issue はマイルストーン単位で立てる。
- 納期軸: どのリリース区切りで出すか（縦スライス）。コンポーネントを横断する。

GitHub の機能には次の制約・性質がある。

- Sub-issue の親は 1 つだけ。1 つのタスクを「コンポーネント」と「納期スライス」の両方の子に同時には置けない。
- ネイティブ Milestone は、Sub-issue の親子とは独立に付与でき、コンポーネントを横断できる。期日と進捗バー（closed/total）を持つ。

この 2 軸を 1 つの階層に押し込めると、どちらかの見え方が壊れる。

## 決定

2 軸をそれぞれ別の GitHub 機能に割り当てる。

- 構造軸 → 親 issue ＋ Sub-issues（2 段。深くしない）
- 納期軸 → ネイティブ Milestone

各タスク（子 issue）は「親 = コンポーネント」と「Milestone = 納期スライス」の交点に置く。

```text
Milestone: M1 …（納期スライス, due 付き）
  ├ 子 issue …   ← 親: [componentA]
  ├ 子 issue …   ← 親: [componentB]
  └ 子 issue …   ← 親: [componentC]
```

### 各レベルに書くこと

#### Milestone（納期スライス）

- タイトル: 達成したら何が動くかが分かる名前
- 期日（due date）
- ゴール: このスライスが通ったら確認できる状態（例: end-to-end の疎通）
- スコープ: 含む / 含まない

#### 親 issue（コンポーネント）

- ゴール: このまとまりで達成したい状態
- スコープ: 含む / 含まない
- サブタスク: 子 issue 番号の控え（実体は Sub-issues で紐付け）
- 関連: 依存コンポーネント・参考

#### 子 issue（サブタスク, 1 PR 粒度）

- 親 issue へのリンク
- 背景 / コンテキスト: なぜ今やるか、関連する issue / spec / ADR へのリンク（diff から読めない前提）
- 内容: やること。対象クレート・ファイルや想定する型 / API まで具体的に
- スコープ外: この PR でやらないこと
- 完了条件: チェックリスト（検証コマンドや観察可能な状態を含める）
- 設計メモ / 論点: 設計判断・注意点
- Milestone: 納期スライスを付与（サイドバー）

入力の雛形は `.github/ISSUE_TEMPLATE/`（`parent.md` / `subtask.md`）に置く。

## 結果/影響

- コンポーネント進捗（親 issue の Sub-issue ロールアップ）と、リリース進捗（Milestone の %）を独立に見られる。
- Sub-issue 階層を 2 段に保つことで、ボード表示と進捗集計が追いやすい。

## 注意点

- 1 タスクは 1 親（コンポーネント）に属する。コンポーネント横断は Milestone 側で表現する。
- 親 issue はマイルストーンごとに立て、sub-issue が全て閉じたらクローズする。次のマイルストーンで同じコンポーネントを扱うときは新しい親を立てる。
