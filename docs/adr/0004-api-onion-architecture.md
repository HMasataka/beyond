# 0004. API はオニオンアーキテクチャと機能別パッケージで構成する

- ステータス: Accepted
- 日付: 2026-06-22

## コンテキスト

API は `api/internal/` 配下に handler・usecase・repository・di・openapi・migrations のディレクトリを持つ。
これらをどう関係づけ、機能をどこに実装するかの規約がないと、次の問題が起きる。

- ビジネスロジックが handler に漏れ、入力（HTTP）と業務の関心が混ざる。
- 依存の向きが定まらず、上位と下位が双方向に参照して循環や密結合を招く。
- 1 つの層パッケージに全機能のファイルを平置きすると、機能が増えるほど肥大化し、変更が衝突しやすい。

依存方向と、機能ごとのファイルの置き場所を、あらかじめ規約として決めておきたい。

## 決定

オニオンアーキテクチャを採用する。層を同心円で捉え、依存は外側から内側への一方向に限る。
中心は業務の関心（ドメイン）で、外側ほど技術的詳細（HTTP・DB）に近い。

層は `internal/` の直下に置き、機能が増える層では、各層の下に機能ごとのパッケージを作って実装する。

```text
internal/
├── domain/        ドメイン層（中心）。エンティティと、外側が実装するインターフェース
│   └── <機能>/
├── usecase/       アプリケーション層。ユースケースを組み立てる。domain にのみ依存
│   └── <機能>/
├── repository/    永続化の実装。domain のインターフェースを満たす
│   └── <機能>/
├── handler/       HTTP 入力。openapi 経由で受け、usecase を呼ぶ
│   └── <機能>/
├── di/            合成ルート。具体実装を結線する
├── openapi/       OpenAPI からの生成物（編集しない）
└── migrations/    マイグレーション
```

`<機能>` は対象の業務領域ごとのパッケージ名を表すプレースホルダで、`user`・`order` のように具体名を入れる。
たとえばユーザー機能なら `internal/usecase/user/`・`internal/repository/user/` のように、各層の下に同名のパッケージを作る。

### 依存のルール

- 中心の `domain` は他のどの層も import しない。
- `usecase` は `domain` にのみ依存する。
- `handler` は `usecase`（と入力の契約である `openapi`）に依存する。
- `repository` は `domain` のインターフェースを実装し、`domain` に依存する。
- インターフェースは内側で定義し、外側が実装する（依存性逆転）。これにより `usecase` は永続化の実装を知らずに済む。
- `di` だけが各層を import してよい。具体実装を選んで結線する合成ルートとして、依存方向のルールの外に置く。

### 機能を追加するとき

新しい機能は、関係する層の下に同名のパッケージを足して実装する。
層をまたいで 1 つの機能を仕上げる形になり、既存機能のパッケージには触れない。

### 生成された単一インターフェースを集約して満たす

OpenAPI から生成されるサーバインターフェースは、全オペレーションを 1 つにまとめた `openapi` パッケージの型である。
これを機能別の handler パッケージに分けるため、各機能は担当するオペレーションだけを実装した型を持ち、集約型 `handler.Server` がそれらを匿名フィールドとして埋め込む。
埋め込みでメソッドが昇格し、`Server` が生成インターフェース全体を満たす。

匿名フィールド名は型名になるため、機能ハンドラの型名は機能ごとに一意にする（`health.Health` のように機能名を型名にする）。
機能ハンドラが依存（usecase など）を持つ場合は、生成時に渡して結線する。

## 結果

- 依存が内向きの一方向に揃い、業務ロジック（`domain`・`usecase`）が HTTP や DB の詳細から独立する。
- インターフェースを内側に置くため、テストでは外側の実装を差し替えられ、`usecase` を単体で検証できる。
- 機能の追加は各層に同名パッケージを足すだけになり、既存ファイルとの衝突を避けられる。空のままだった `usecase`・`repository`・`di`・`migrations` に役割が定まる。
- 却下した代替案: 機能優先（feature-first）で `internal/<機能>/` の下に handler・usecase・repository を同居させる案。機能内の凝集は高まるが、層の依存方向がパッケージ境界に現れず、内向き一方向の制約を機械的に保ちにくいため採らない。
- 制約: 層をまたぐ参照は内向きのみとする。外側を内側から参照したくなったらインターフェースを内側に定義して逆転させる。
