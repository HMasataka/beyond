# 0007. オニオンの依存方向を depguard で強制する

- ステータス: Accepted
- 日付: 2026-06-23

## コンテキスト

ADR 0004 でオニオンアーキテクチャを採り、依存は外側から内側への一方向に限ると決めた。
ただし規約を文章で示すだけでは、層をまたぐ誤った import（usecase が repository を直接呼ぶ、domain が openapi を参照するなど）を防げない。レビューでの目視に頼ると、機能が増えるほど抜けやすい。

依存方向を機械的に検査し、違反を CI で落としたい。
golangci-lint は既に lint に組み込んである（ADR 0003 の Taskfile から `lint:api` で実行）。
依存検査の手段は、golangci-lint 同梱の depguard と、独立ツールの go-arch-lint が候補になる。

## 決定

依存方向の強制に depguard を採用し、`api/.golangci.yml` にルールを定義する。

- 方式は denylist とする。各層の `files` に対し、内向き一方向に反する「外向き」の import パッケージを `deny` に列挙する。stdlib や外部ライブラリは制約しない。
- ルールは層ごとに置く。`domain` は他の全層を、`usecase` は domain 以外を、`handler` は repository・infra など、`repository` は上位層を、`infra` は業務層を、それぞれ拒否する。
- `di` にはルールを置かない。具体実装を結線する合成ルートとして全層を import してよい（ADR 0004 の例外規定）。
- `openapi` は生成物のためルール対象にしない。handler から使うのは許可する。
- 各 `deny` の `desc` に ADR 0004 を参照する文言を入れ、違反時のメッセージから意図を辿れるようにする。

## 結果

- 層をまたぐ誤った依存が `task lint` と CI で落ちる。レビューの目視に頼らず、ADR 0004 の依存ルールが機械的に守られる。
- 違反メッセージに「なぜ禁止か」が出るため、ルールに当たった開発者が ADR 0004 まで辿れる。
- 機能が空の現状では何も検出しないが、各層に機能パッケージを足すと同時にルールが効く。テンプレートの段階から制約が備わる。
- denylist のため、新しい層や移動が起きたときはルールの追従が要る。許可漏れで誤検出が出るより、禁止漏れで検出を取りこぼすリスクの方が残る。
- 却下した代替案: go-arch-lint。レイヤ定義を宣言的に書けて表現力は高いが、ツールと設定ファイルが別に増える。既存の golangci-lint に乗る depguard なら追加の実行系統が要らず、テンプレートの土台を軽く保てる。
