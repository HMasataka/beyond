# ADR（アーキテクチャ決定記録）

アーキテクチャや設計上の意思決定を記録する。
diff やコードからは読み取れない「なぜこうしたか」を残し、後から決定の経緯を辿れるようにする。

## 書くタイミング

次のいずれかに当てはまる決定をしたら ADR を起こす。

- 代替案があって、その中から一つを選んだとき
- 制約（外部 API の仕様、依存関係、運用条件など）によって設計の形が決まったとき
- 後から「なぜこうしたか」を知りたくなる、影響範囲の広い決定をしたとき

軽微な実装上の選択や、コードを読めば自明な判断は対象にしない。

## ファイルの置き方

- 置き場所: このディレクトリ（`docs/adr/`）
- ファイル名: `NNNN-タイトル.md`。`NNNN` は 4 桁の連番で、既存の最大番号の次を採る
- タイトルは決定の対象が分かる短い句にする

## 構成

各 ADR は次のセクションで書く。

- ステータス: Proposed / Accepted / Superseded のいずれか
- 日付: 決定した日（YYYY-MM-DD）
- コンテキスト: 前提と制約。何が問題か
- 決定: 採用した方針
- 結果: 決定によって得られるものと、残る制約やトレードオフ

却下した代替案は、理由を添えて「コンテキスト」または「結果」に残す。
後から同じ案を蒸し返さないためで、理由のない却下は記録の価値がない。

## 決定が覆ったとき

古い ADR は消さない。
ステータスを Superseded に変え、新しい ADR から旧 ADR を参照する。

## 文章の書き方

ADR を書くときは `japanese-tech-writing` スキルの規範に従う。

## 雛形

```markdown
# NNNN. タイトル

- ステータス: Accepted
- 日付: YYYY-MM-DD

## コンテキスト

（前提と制約。何が問題か）

## 決定

（採用した方針）

## 結果

（得られるもの、残る制約やトレードオフ。却下した代替案とその理由）
```
